# 高级树结构

## 定义
高级树结构是在基础二叉树、二叉搜索树基础上的优化和扩展，通过巧妙的平衡规则和存储策略，确保在数据量巨大或操作频繁时，依然能保持卓越的性能。

主要包括：
- **AVL 树**：高度平衡的自平衡二叉搜索树
- **红黑树**：工程实践中广泛使用的自平衡树
- **B 树**：多路平衡搜索树，用于磁盘存储
- **B+ 树**：B 树的升级版，数据库索引的标准

## 原理

### 为什么需要高级树结构？
普通二叉搜索树（BST）在插入有序数据时会退化成链表，查找性能从 O(log n) 恶化到 O(n)。高级树结构通过自动调整树的形态，保持树的平衡，确保操作的时间复杂度始终维持在 O(log n) 级别。

### AVL 树：高度平衡的守护者
AVL 树是最早被发明的自平衡二叉搜索树。

**核心概念：平衡因子**
- 平衡因子 = 左子树高度 - 右子树高度
- 在 AVL 树中，每个节点的平衡因子只能是 -1、0 或 1
- 当平衡因子变为 2 或 -2 时，树失衡，需要旋转调整

**四种旋转操作**：
1. **左旋**：处理"右右"失衡情况
2. **右旋**：处理"左左"失衡情况
3. **左右旋**：先左旋后右旋，处理"左右"失衡
4. **右左旋**：先右旋后左旋，处理"右左"失衡

**特点**：
- 严格平衡：平衡条件严格（高度差≤1），查找性能极佳
- 调整频繁：插入/删除后可能需要多次旋转
- 适用场景：查询多，增删少的场景

### 红黑树：工程实践的王者
红黑树通过为节点增加颜色属性（红色或黑色）并遵守五项规则来维持平衡。

**五项核心规则**：
1. 每个节点非红即黑
2. 根节点是黑色的
3. 所有叶子节点（NIL 空节点）都是黑色的
4. 红色节点的两个子节点必须是黑色的（不能有连续的红色节点）
5. 从任一节点到其每个叶子节点的所有路径都包含相同数目的黑色节点

**插入调整策略**：
- 插入新节点时总是先着为红色
- 如果导致冲突，通过变色和旋转调整
- 将冲突不断向树根方向"上移"，直到解决

**特点**：
- 宽松平衡：通过颜色规则保证大致平衡
- 增删效率高：旋转次数较少
- 适用场景：语言标准库（Map/Set）、文件系统

### B 树与 B+ 树：磁盘存储的引擎
当数据量太大，无法全部装入内存时，必须存放在磁盘上。B 树和 B+ 树为了减少磁盘 I/O 次数而设计。

**m 阶 B 树的特性**：
- 每个节点最多有 m 个子节点
- 每个非根、非叶子节点至少有 ⌈m/2⌉ 个子节点
- 根节点至少有 2 个子节点（除非是叶子节点）
- 所有叶子节点都位于同一层
- 一个非叶子节点如果有 k 个子节点，则包含 k-1 个键

**B+ 树与 B 树的核心区别**：
1. **数据存储位置**：B+ 树中，所有数据记录只存储在叶子节点中，内部节点仅存储键用于路由
2. **叶子节点链表**：所有叶子节点通过指针串联成有序双向链表

**B+ 树的优势**：
- 更稳定的查询效率：任何查找都必须走到叶子节点，路径长度相同
- 更高的空间利用率：内部节点不存数据，能容纳更多键
- 强大的范围查询：通过叶子节点的链表高效进行范围查询
- 更适合磁盘预读：节点通常设计为恰好一页大小

## 示例

### AVL 树节点定义
```python
class AVLNode:
    def __init__(self, key):
        self.key = key
        self.height = 1  # 节点高度
        self.left = None
        self.right = None

def get_height(node):
    return node.height if node else 0

def get_balance_factor(node):
    if not node:
        return 0
    return get_height(node.left) - get_height(node.right)

def right_rotate(y):
    x = y.left
    T3 = x.right
    
    # 执行旋转
    x.right = y
    y.left = T3
    
    # 更新高度
    y.height = 1 + max(get_height(y.left), get_height(y.right))
    x.height = 1 + max(get_height(x.left), get_height(x.right))
    
    return x
```

class RBNode:

## 易错点
1. **AVL 树**：
   - 平衡因子计算错误
   - 旋转顺序搞混（左右旋 vs 右左旋）
   - 更新高度的顺序错误

2. **红黑树**：
   - 颜色规则记忆混淆
   - 插入后修复逻辑复杂
   - 需要父节点指针

3. **B 树/B+ 树**：
   - 节点分裂和合并规则复杂
   - B 树和 B+ 树的区别混淆
   - 阶数 m 的含义理解错误

## 复杂度与考点

### 性能对比
| 树结构 | 查找 | 插入 | 删除 | 平衡性 |
|--------|------|------|------|--------|
| AVL 树 | O(log n) | O(log n) | O(log n) | 严格平衡 |
| 红黑树 | O(log n) | O(log n) | O(log n) | 大致平衡 |
| B 树 | O(log_m n) | O(log_m n) | O(log_m n) | 多路平衡 |
| B+ 树 | O(log_m n) | O(log_m n) | O(log_m n) | 多路平衡 |

### 典型应用
| 树结构 | 典型应用 |
|--------|----------|
| AVL 树 | 内存中需要快速查找的场景 |
| 红黑树 | Java TreeMap/TreeSet、C++ STL map/set、Linux 内核 |
| B 树 | 早期文件系统、某些数据库索引 |
| B+ 树 | MySQL、PostgreSQL 等现代关系型数据库索引 |
